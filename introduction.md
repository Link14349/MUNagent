# 总体概况
这是基于LLM实现的自动化模拟联合国场景设计&推演工具. 主要针对场景是模拟联合国历史委员会(危机联动).

使用python语言, agent模型调用Deepseek v4 flash/pro的api接口(但是也可以选择使用其他模型, 自行设置模型调用的url以及api key, 不过默认的api调用url用deepseek的; 由于DeepSeek接口兼容OpenAI格式, LLM调用层统一按OpenAI兼容协议实现即可), 交互方式为网页GUI.

工具的两大核心能力:
1. **会场设计**: 从一个历史主题出发, 由设计Agent辅助用户生成完整的会场设定(背景、会场结构、国家/人物席位、危机弧线、数值体系等), 最终导出为一个可复用的"场景包".
2. **会场推演**: 加载场景包后, 由代表Agent扮演各席位进行发言、磋商、写指令, 由推演Agent(主席团)控制会议进程并判定指令结果, 完成一次完整的危机联动推演. 支持纯AI推演(观察模式)与人类接管部分席位的混合模式.

# 设计功能
主要分为两种: 会场设计, 会场推演. 会场设计是用来从头开始设计一个会场的背景信息、国家、人物等设定等. 会场推演则是一个代表扮演一个agent, 进行行动/发言等操作, 然后进行推演.

## 会场设计

### 设计流程
会场设计采用"向导式 + 对话式"结合的流程, 由一个**设计Agent**辅助用户逐步完成. 每一步设计Agent先生成草稿, 用户可以直接编辑或通过对话让Agent修改, 确认后进入下一步. 各步骤之间有依赖关系(比如席位设计依赖会场结构), 但允许回退修改(回退后下游内容标记为"需要复查").

设计步骤如下:

1. **主题与历史切入点**: 用户给出主题(如"法国1848革命"、"古巴导弹危机"), 设计Agent生成若干候选切入时间点与危机起始状态, 用户选定. 确定推演的起始日期与(可选的)预期结束条件.
2. **资料检索与整理**: 在生成背景文书之前, 设计Agent围绕选定的主题与切入点智能联网搜索相关历史资料(专著章节、论文、档案、年表等), 下载有价值的文件. 对PDF等非文本格式, 调用**在线MinerU服务**转换为Markdown后再阅读(调用方式见[docs/tools/agent-api-pdf-to-markdown-guide.md](docs/tools/agent-api-pdf-to-markdown-guide.md), 同步接口适合小文件、异步接口适合整本书; 服务地址作为项目配置/环境变量读取, 见"配置与密钥管理"; 本地部署MinerU留作后续选项, v1只做在线服务). 整理后的资料统一存放在场景包的`references/`目录中, 每份资料附带来源元信息(URL、获取日期、原始文件名). 用户可以在此步骤审阅资料清单、手工增删.
3. **背景文书(Background Guide)**: 基于`references/`中的参考资料生成危机的历史背景、各方立场综述、核心矛盾与议题清单. 这份文书同时服务于两个用途: 给人类玩家阅读, 以及作为所有Agent的公共知识基础.
4. **会场结构设计**: 确定是单一总会场还是多分会场(如苏联政治局/美国国会), 每个会场的性质、决策规则(投票门槛、是否有一票否决等)、初始议程.
5. **席位设计**: 为每个会场设计席位(历史委中一般是具体历史人物). 每个席位包含:
   - 公开信息: 姓名、职务、所属阵营、公开立场;
   - 私密信息: 秘密目标(个人野心、隐藏议程)、私人关系网、初始私有资源;
   - 权力清单(Portfolio Powers): 该角色凭个人身份可以直接动用的权力(如"陆军部长可以调动XX军区的部队"), 这是个人指令判定的依据;
   - 人格卡(Persona Card): 供代表Agent扮演用的性格、说话风格、决策倾向描述.
6. **危机弧线设计**: 预设一条主线危机弧(若干个按时间/条件触发的预埋事件, 如"若推演至第3天仍未解决XX问题, 则爆发XX事变"), 以及一组可选的随机事件池. 弧线只是DM的参考剧本, 推演中DM可以根据代表行动偏离它.
7. **数值体系设定(可选, 建议轻量)**: 为主要阵营/角色设定少量关键数值(如军事力量、经济资源、民意/威望), 作为DM判定指令成败时的参考依据, 而不是精确的战棋数值. 数值的粒度在设计时可配置(无数值/粗粒度标签/数值).
8. **审查与导出**: 设计Agent对整个场景包做一致性检查(席位权力与会场结构是否矛盾、秘密目标是否互相冲突到无法推演等), 生成检查报告, 用户确认后导出场景包.

### 场景包结构
场景包是一个自包含的目录(或zip), 全部为人类可读可编辑的文件, 便于手工微调与社区分享:

```
scenario/
├── manifest.yaml        # 元信息: 名称、作者、版本、起始时间、简介与梗概
├── background.md        # 背景文书(公共知识)
├── venues.yaml          # 会场定义: 结构、决策规则、初始议程、席位清单(seats)
├── seats/               # 每个席位一个文件
│   └── <seat_id>.yaml   # 公开信息 + 私密信息 + 权力清单 + 人格卡
├── crisis_arcs.yaml     # 危机弧线: 预埋事件 + 随机事件池 + 推演终局条件
├── stats.yaml           # 数值体系(可选)
└── references/          # 参考资料(设计阶段检索整理的历史资料)
    ├── index.yaml       # 资料清单: 每份资料的来源URL、获取日期、原始文件名
    ├── <doc_id>.md      # 转换/整理后的Markdown正文
    └── raw/             # (可选)原始文件(如PDF), 默认不随场景包分享以控制体积
```

信息可见性在文件层面就要分开: `background.md`和席位的公开信息是全体可见的; 席位私密信息只注入对应代表Agent的上下文; `crisis_arcs.yaml`只有DM和主席可见.

## 会场推演

### 推演运行架构
推演程序由三个部分组成: **会议状态机**(管"会议怎么开")、**推演Agent即主席团**(管"会议由谁控制、行动如何裁决")、**代表Agent**(管"谁在开会"). 三者的关系是: 状态机提供会议的骨架与规则, 代表Agent在骨架内行动, 主席团裁决行动结果并推动骨架前进.

#### 会议状态机
负责会议组织与管理. 首先是**会场形态**, 历史委员会可能有三种:

- **单一总会场**: 如法国1848革命危机联动, 就是一个总的法兰西国民议会会场;
- **多分会场**: 如古巴导弹危机, 苏联政治局一个分会场, 美国国会一个分会场;
- **临时联合会场**: 分会场情况下, 由若干代表(或几个分会场的全部代表)临时组成的联合会议.

**交互边界规则**: 只有处于同一个会场的代表才能直接交流会话、投票表决; 跨会场的交互只能通过提交指令给DM, 由推演Agent间接传达; 若需要面对面谈判, 可以临时将参与谈判的代表单独开到一个临时分会场中进行.

在此基础上, 状态机的职责是: 组织、更新各个会场内的会议节奏(Crisis Update, 自由发言/主持核心磋商(Moderated Caucus), 非正式磋商, 指令提交, 投票表决等), 并管理分会场间经由推演Agent的交互.

#### 推演Agent(主席团)
控制会议进程与会场间交互, 由三个角色组成:

- **会议主席**: 整体的会议控制管理——控制会议进度(如决定从自由发言/主持核心磋商切换到非正式磋商、结束议程、临时组建临时分会场等)、推进时间、给各分会场播报Crisis Update;
- **DM**: 处理各代表提交的指令——进行推演(有随机性的事件通过随机数判定结果)、撰写结果、提交结果给会议主席;
- **书记**: 撰写整个会场的记录, 按时间节点与发生顺序进行概括记录.

#### 代表Agent
扮演会议中的代表(人或国家, 历史委中一般是具体的历史人物), 也就是用来开会的. 可以与会场内的代表进行磋商、投票、写指令. 指令分为四种:

| 指令类型 | 说明 |
|---|---|
| 联合指令(Directive) | 不需要正式辩论就能通过的行动指令, 格式相对自由, 重"可执行性"而非法律语言 |
| 个人指令(Personal Directive) | 只代表你个人身份采取的行动, 通常私下递交 |
| 公报/声明(Communiqué) | 对外发布的官方表态 |
| 危机笔记(Crisis Note) | 给幕后或其他角色的私信 |

#### 指令处理流水线
所有指令(含分会场间的交互)走同一条统一流程:

```
代表提交个人指令 / 会场投票通过后联合提交指令
  → DM推演判定(随机事件用随机数裁定)并撰写结果
  → 结果提交主席, 主席处理并更新局势
  → 交书记记录
  → 主席以Crisis Update播报给各分会场(每个分会场得到的Crisis Update文本可以不同)
    (播报可以延迟, 也可以对某个分会场扣住不发)
```

### 会议状态机与回合调度
会议采用**前场/后场双轨模型**: 前场是真正的状态机(辩论流程), 后场是一条持续运行的指令流水线; Crisis Update不是状态, 而是可以抢占前场任意阶段的中断事件. 这样"临时突发指令"就有了自然的归宿——个人指令和危机笔记随时写随时交, 走后场队列, 不经过前场状态机.

**前场轨(状态机)**: 每个会场独立持有一个状态机实例:

```
ModeratedCaucus(主持核心磋商) ⇄ UnmoderatedCaucus(非正式磋商)
              ⇅
   Voting(投票表决: 动议触发的子流程)
```

- 阶段切换由主席Agent决定(依据会场氛围、代表动议、议程进度、以及可配置的阶段轮数上限), 人类导演(见"人类参与模式")可以随时干预强制切换;
- Voting不是常驻阶段, 而是由"动议表决某联合指令"触发的子流程, 可以随时插在辩论中间, 表决完成后返回被打断的阶段.

**后场轨(指令流水线)**: 不是状态, 而是一条持续收活的队列, 与前场并行运行. 个人指令与危机笔记在任何前场阶段都可以随时递交, 直接进入后场队列; 联合指令在投票通过后进入同一队列. DM持续消费队列: 合法性检查→推演判定→撰写结果→上报主席(即"指令判定机制"一节的流水线).

**Crisis Update(中断事件)**: 由主席决定插入的广播, 可以抢占前场任何阶段. 触发源包括: 已通过指令的执行后果酿成新事态、危机弧线预埋事件到达触发点、个人指令/危机笔记搞出大动静、或纯粹为了会议节奏(辩论僵局时后场投放新危机搅局). 播报完毕后由主席决定返回被打断的阶段还是切换到新阶段.

**回合调度与时钟**:

- LLM推演本质上是回合制的: 在ModeratedCaucus中, 主席按发言名单点名, 每轮被点到的代表Agent产出一次发言. 代表Agent的"随时递交个人指令/危机笔记"在实现上挂在其行动回合里: 每次轮到某代表行动时, 其可选动作除发言/动议外还包括写指令;
- **UnmoderatedCaucus采用"小轮+屏障"的动态分组模型**, 允许代表在磋商过程中换组:
  - 进入非正式磋商时, 代表Agent自主表达"想找谁磋商", 系统据此分成若干小组; 整个非正式磋商切分为若干**小轮(mini-round)**, 轮数可配置;
  - **轮内**: 各小组并行推进一轮对话(asyncio并发调LLM, 每个成员发言一次左右). 轮内成员表冻结, 因此并行没有竞态;
  - **轮间屏障**: 每个代表在本轮最后一次发言的结构化输出中附带去留决策(留下/转去某组/拉人新开一组/单独行动如埋头写危机笔记), 不需要额外的LLM调用. 引擎在屏障处按固定席位顺序原子地结算所有移动, 写入事件日志(保证回放可复现), 然后开启下一轮;
  - 组的生灭自然涌现: 成员全部离开即消亡, 多人转投同一目标即合流; 小组数量设上限, 防止碎成一堆单人组;
  - "某代表加入小组"本身是组内可见事件, 组内其他人下一轮看到后可以自然反应(换话题、警惕、驱逐性冷场);
  - 中途加入者看不到加入前的组内聊天: group scope的事件在发出时就物化"当时的成员列表"作为可见名单, 这一点由事件模型天然实现, 无需额外逻辑;
  - **闭门小组(door-closed)**: 小组发起人可将小组标记为"闭门". 外人申请加入闭门小组时, 引擎在屏障结算中向发起人Agent发起一次轻量决策调用(放行/拒绝), 拒绝后申请者留在原组(或单独行动)并知晓自己被拒, 组内成员也能看到"有人想进来被拒了"这一事件;
  - 玩家模式下人类的换组/申请加入请求同样在下一个屏障统一生效, 引擎逻辑不分叉;
- **推演时钟**: 状态机维护"故事内时间"(如1962年10月16日 14:00), 与真实时间解耦. 主席在Crisis Update时推进故事时间; 指令的生效也挂在故事时间轴上(如"部队调动需要12小时"), DM据此裁定时序冲突(谁的行动先生效);
- 每个阶段有可配置的预算上限(最大发言轮数/最大LLM调用次数), 防止推演失控地消耗token.

### 信息可见性与通信模型
这是危机联动的核心机制, 所有消息与记录都必须带**可见范围(scope)**:

| 范围 | 说明 | 例子 |
|---|---|---|
| global | 全体会场可见 | 公报、已播报的Crisis Update |
| venue | 单一会场内可见 | 会场发言、会场投票结果 |
| group | 临时磋商小组内可见 | 非正式磋商内容 |
| private | 指定席位+主席团可见 | 危机笔记、个人指令、秘密目标 |
| dm-only | 仅主席团可见 | 危机弧线、未播报的推演结果 |
| self | 仅行为者本席位可见 | 内心动机(`speech_thought`), 主席团与上帝视角(复盘)可见 |

实现上用一条统一的**事件总线**: 所有行为(发言、指令、判定结果、阶段切换)都是一个带scope的事件, 追加写入事件日志. 每个Agent构建上下文时只能读取自己scope内的事件. 这个事件日志同时就是推演的存档格式——存档/回放/断点续推都基于它(事件溯源).

书记Agent的记录也按scope分层: 会场公开记录(venue)、主席团内部全记录(dm-only). 人类观察者界面上可以选择"上帝视角"(看全部)或"某席位视角"(只看该席位可见的).

### 指令判定机制
DM判定一条指令的流程:

1. **合法性检查**: 指令是否在提交者的权力清单(Portfolio Powers)范围内? 联合指令是否通过了所需的投票门槛? 越权的指令直接驳回并说明理由.
2. **可行性评估**: DM结合当前局势、双方数值/实力标签, 给出该行动的成功概率档位(如: 稳操胜券90%/占优70%/五五开50%/冒险30%/异想天开10%).
3. **随机判定**: 由**程序**(而非LLM)掷随机数对照概率档位, 得到结果档位(大成功/成功/部分成功/失败/灾难性失败). 随机数种子记录进事件日志, 保证回放可复现.
4. **结果撰写**: DM根据结果档位撰写事件结果叙述, 更新相关数值/状态, 标注哪些部分对哪些会场可见、是否延迟播报.
5. **上报主席**: 主席决定播报节奏(立即Crisis Update/延迟/扣住不发), 交书记记录.

关键原则: LLM负责"评估概率"和"叙述结果", 程序负责"掷骰子", 避免LLM既当运动员又当裁判导致的叙事偏好(LLM倾向于让戏剧性的事情总是成功).

### 代表Agent的记忆与上下文管理
长时间推演会远超单次上下文窗口, 每个代表Agent的上下文按如下结构组装:

1. **固定层**: 人格卡 + 席位公开/私密信息 + 背景文书摘要 + 指令格式说明;
2. **长期记忆层**: 书记维护的会场公开记录摘要(滚动摘要, 旧内容逐步压缩) + 该席位的私人记忆摘要(自己参与过的磋商、递过的危机笔记、收到的私信的滚动摘要);
3. **近期原文层**: 最近N条该席位可见的事件原文;
4. **当前任务层**: 当前阶段说明 + 本次要做的事(发言/投票/写指令)及输出格式要求.

代表Agent每次行动的输出采用结构化格式(如JSON: 行动类型+内容+简短的内心动机), 内心动机不进入其他代表的可见范围, 但供DM理解意图、供复盘界面展示.

### 人类参与模式
推演支持三种参与形态, 可混合:

1. **观察模式**: 全AI推演, 人类只观看(可选任意视角), 可以暂停/回放;
2. **导演模式**: 人类扮演主席团的一员或全部(最常见的是人类当DM或主席, 其余由AI辅助草拟、人类修改后发出), 也可以在AI主席团运行时随时下达干预(强制切阶段、注入自定义危机事件);
3. **玩家模式**: 人类接管一个或多个代表席位, 通过GUI进行发言、磋商、投票、写指令, 其余席位由AI扮演. 人类回合设有可配置的等待超时(超时后AI代打或跳过), 避免卡住整场推演.

# 系统架构

## 技术选型
- **后端**: Python 3.11+, FastAPI + WebSocket(推演事件实时推送到前端), asyncio驱动Agent并发;
- **前端**: Vue3 + Vite + TypeScript SPA(单独目录, 构建产物由FastAPI静态托管), 核心页面见"网页GUI设计";
- **持久化**: SQLite(单文件, 零部署成本). 核心表就是事件日志表 + 场景包索引 + 推演会话表;
- **LLM调用层**: 统一OpenAI兼容客户端封装, 按Agent角色配置模型(建议: 代表/书记用`deepseek-v4-flash`, 主席/DM/设计用`deepseek-v4-pro`), 模型配置采用"Provider档案+角色路由"两层结构(见"配置与密钥管理"). 内置重试、超时、token用量统计(推演界面实时显示本场已消耗token/预估费用);
- **Agent框架**: 不引入LangChain等重框架, 自行实现轻量Agent循环(prompt组装→调用→结构化输出解析→校验重试). 本项目的复杂度在"编排"(状态机+事件总线)而不在单Agent推理, 重框架收益小、调试成本高.

## 配置与密钥管理

### 分层配置
配置加载优先级: **环境变量 > 用户配置文件 > 内置默认值**.

- **用户配置文件**: `~/.munagent/config.yaml`, 加入`.gitignore`, GUI设置页的读写目标就是它. 写入时将文件权限设为`600`(仅本用户可读). 单机单人场景下, 本地明文+权限控制是合理的安全线, 不做密钥加密(加密后解密密钥的存放又是同一个问题);
- **环境变量**(如`MUNAGENT_API_KEY`/`MUNAGENT_BASE_URL`): 优先级最高, 方便开发调试和不想让key落盘的用户, 用pydantic-settings加载;
- 后续可选升级: 接`keyring`库存进系统钥匙串(macOS Keychain等), v1不做.

### Provider档案 + 角色路由
若每个Agent角色各存一份key, 同一个key会重复多份、换key要改多处. 因此拆成两层: key只出现在provider档案中, 角色只引用provider名:

```yaml
providers:            # key只在这里出现
  deepseek:
    base_url: https://api.deepseek.com
    api_key: sk-xxxx
  my-local:
    base_url: http://localhost:11434/v1
    api_key: none

roles:                # 角色只引用provider名, 不碰key
  delegate: { provider: deepseek, model: deepseek-v4-flash }
  dm:       { provider: deepseek, model: deepseek-v4-pro }
  chair:    { provider: deepseek, model: deepseek-v4-pro }

tools:                # 外部工具服务
  search:             # 资料检索, 默认Tavily(见02§2)
    provider: tavily
    api_key: tvly-xxx
  mineru:             # 在线MinerU(PDF转Markdown), 供设计Agent的资料检索使用
    base_url: http://36.139.151.129:8282   # 环境变量 MUNAGENT_MINERU_URL 可覆盖
```

注意: MinerU服务当前无鉴权, 其地址应指向内网/VPN环境, 调用前先打其`/health`接口确认可用(详见[docs/tools/agent-api-pdf-to-markdown-guide.md](docs/tools/agent-api-pdf-to-markdown-guide.md)).

### 安全红线
1. **key永远不回传前端**: 设置页展示时只给掩码(`sk-****last4`), 后端API只接受写入、不提供读取完整key的接口——网页GUI意味着存在HTTP边界, 哪怕是localhost也按此纪律执行;
2. **key永远不进事件日志**: 事件日志就是存档格式, 是设计给回放和分享的. LLM调用层要保证请求元数据落日志前剥掉key, 错误信息(如401异常)落日志前做脱敏, 否则一次报错就会把key写进可分享的存档;
3. **场景包和存档导出不含任何配置**: 场景包是社区分享的载体, 导出逻辑与配置系统在代码上不产生交集.

### 连接测试
GUI设置页提供"测试连接"按钮: 后端用当前配置发一次最小请求(如1 token的补全), 当场反馈key是否有效、base_url是否可达, 避免推演进行到一半才发现配置错误.

## 模块划分
按功能域切分: 一层共享地基(config/security/llm) + 两个厚薄不均的子系统(designer 轻、deducer 重). designer 与 deducer 互不依赖, 都只依赖共享层.

```
munagent/
├── config/              # 分层配置加载(共享)
├── security/            # key脱敏等安全卫生(共享)
├── llm/                 # OpenAI兼容客户端、流式增量、模型路由、用量统计(共享)
├── designer/            # 设计子系统
│   ├── scenario/        #   场景包库: 磁盘 CRUD、单文件编辑、历史、chats
│   │   ├── package.py   #     元信息/加载/校验/保存
│   │   ├── files.py     #     单文件操作与文件树
│   │   ├── history.py   #     .history 版本快照
│   │   └── chats.py     #     设计对话 JSONL 持久化
│   ├── agent.py         #   场景设计 Agent loop(function calling + 流式)
│   └── tools/           #   设计工具: 文件读写、联网搜索(Tavily)、下载、MinerU
├── deducer/             # 推演子系统
│   ├── core/            #   事件溯源核心
│   │   ├── events.py    #     事件模型与事件总线(scope过滤、持久化)
│   │   ├── bus.py       #     事件总线
│   │   ├── reducer.py   #     RuntimeState + 事件折叠(续推重建)
│   │   ├── state_machine.py #  会场状态机
│   │   ├── clock.py     #     推演时钟(故事内时间, 内部UTC)
│   │   └── render.py    #     事件渲染(纯函数, golden护住)
│   ├── engine.py        #   推演引擎: 驱动状态机、调度Agent回合
│   └── agents/          #   推演期Agent
│       ├── base.py      #     Agent基类: 上下文组装、结构化输出、重试
│       ├── delegate.py  #     代表Agent
│       ├── chair.py     #     主席Agent
│       ├── dm.py        #     DM Agent(含判定流水线)
│       └── recorder.py  #     书记Agent(滚动摘要)
├── server/              # FastAPI: REST(设计/管理) + WebSocket(推演)(共享)
└── web/                 # 前端(Vue3 + Vite + TypeScript)
```

## 核心数据模型(草案)
- **Scenario**: 场景包整体, 见"场景包结构";
- **Session**: 一次推演会话 = 场景包引用 + 配置(参与模式、模型配置、预算) + 事件日志;
- **Event**: `{id, session_id, 故事时间, 真实时间, 类型, 发起者, scope, 可见席位列表, 内容, 随机种子(判定类事件)}` —— 一切皆事件;
- **Venue / Seat**: 运行时状态(当前阶段、在场席位、临时会场的父子关系、席位当前数值);
- **Directive**: 指令实体, 有自己的生命周期状态: 草稿→已提交→(投票中→已通过/已否决)→判定中→已裁决→已播报.

# 网页GUI设计
- **首页**: 场景包库(内置示例 + 用户创建的), 新建设计/开始推演入口;
- **设计工作台**: 左侧步骤导航(七步流程), 中间为当前步骤的结构化编辑表单, 右侧为设计Agent对话栏; 支持任一步骤重新生成/手工编辑;
- **推演大厅**: 推演的主界面, 布局为:
  - 顶部: 故事时间、当前阶段、各会场标签页;
  - 中间: 当前会场的时间线(发言/事件流, 按scope过滤后的视角内容);
  - 右侧: 席位列表(含发言状态)、指令追踪面板(每条指令的生命周期状态);
  - 底部(玩家模式): 人类操作区(发言输入、指令编辑器、投票按钮);
  - 导演模式额外有"主席团控制台": 强切阶段、注入事件、审改AI草稿、查看dm-only信息;
- **复盘页**: 推演结束后按事件日志生成全场回放, 支持任意视角切换、跳转到任意时间点、导出会议记录(markdown).

# 开发路线图
- **M1 - 最小推演内核**: 事件总线 + 单会场状态机 + 代表/主席/DM Agent, 用一个硬编码小场景在CLI里跑通"发言→指令→判定→Crisis Update"闭环;
- **M2 - 完整会议机制**: 全部阶段与指令类型、投票、书记滚动摘要、推演时钟、判定流水线(概率档位+程序掷骰);
- **M3 - 多会场**: 分会场、跨会场交互、临时会场(谈判)、可见性scope完整实现、存档与回放;
- **M4 - Web GUI(推演)**: FastAPI + WebSocket + 推演大厅/复盘页, 观察模式先行, 再加导演/玩家模式;
- **M5 - 会场设计**: 设计Agent + 设计工作台 + 资料检索工具链(联网搜索、下载、在线MinerU转Markdown) + 场景包导入导出与一致性检查;
- **M6 - 打磨**: token预算控制面板、多模型路由配置页、内置1-2个精修示例场景(如古巴导弹危机).

里程碑顺序刻意把"设计"放在"推演"之后: 推演内核跑通之前, 场景包格式会频繁变动, 先用手写的示例场景包驱动开发, 格式稳定后再做设计器.

# 已定决策
- **单机单人**: v1不做多人在线, 按单机单人实现(一人可控多席位), 不需要用户/席位认领机制;
- **数值体系**: 默认走"粗粒度标签+DM自由裁量", 精确数值战棋化作为可选高级模式;
- **欺骗边界**: 允许代表Agent说谎/隐瞒, 但在人格卡中显式给出诚信倾向参数, 避免所有Agent都无底线背刺导致推演失真.
