# 11 - 成本控制与缓存命中

> 上级文档: [index.md](index.md) | 相关: [05-agent-harness.md](05-agent-harness.md)(上下文组装), [03-event-model.md](03-event-model.md)(用量表), [08-config.md](08-config.md)(配置)

## 1. 前提: 前缀缓存的计费模型

DeepSeek(及多数OpenAI兼容服务)为**自动前缀缓存**: 两次请求开头完全一致的部分按缓存价计费(约为正常输入价1/10), 从第一个不同token起全价. 由此得出全部设计的**总原则**:

> 易变内容往prompt尾部放; prompt只能"从尾部生长", 不能"从中间改".

缓存命中率不是调参问题, 是上下文组装的结构问题.

### DeepSeek缓存机制要点(已查证, 2026-07)

- **作用域是账号(user), 不是API key**: 账号间缓存逻辑隔离互不可见; 同账号下所有key共享同一缓存池;
- **内容寻址的前缀块存储**: 从第0个token起严格前缀匹配, 以64 token为存储块, 存于分布式磁盘. 逻辑上是**树**而非线性缓冲——多个Agent以相同G段开头、L1起各自分叉, 各分支是独立缓存条目, **互不覆盖**. 一堆Agent共用一个key不会互相顶掉缓存, 这正是本设计的依赖前提;
- **因此"每个Agent配一个API key"无用甚至有害**: 同账号多key行为与单key完全一致(白增密钥管理成本); 跨账号则隔离掉G段共享, 花更多钱买更差命中率. 维持单key设计;
- **淘汰**: 未使用条目"数小时到数天"自动清除; 推演连续进行, 正常够不着;
- **持久化时机**: 缓存块在请求完成后写入, 且共享前缀可能需观察到两个同前缀请求后才提取为独立缓存单元——这是§6预热设计的依据, 预热后首批并发仍可能零星miss, 属一次性成本.

来源: [DeepSeek官方发布公告](https://api-docs.deepseek.com/news/news0802/) / [官方Context Caching指南](https://api-docs.deepseek.com/guides/kv_cache/) / [命中规则实测指南](https://deepseekv4pro.com/guides/deepseek-context-caching-hit-rules). 若换用其他provider, 缓存语义需重新查证(如OpenAI的自动缓存与路由相关, 可传`prompt_cache_key`提升命中).

## 2. 上下文五段结构(按变化频率递增排序)

替代05中早期的四层方案, 现行结构:

| 段 | 内容 | 变化频率 | 缓存表现 |
|---|---|---|---|
| G 全局共享段 | 会议规则+四类指令格式+全部任务输出schema+背景文书摘要 | 全场不变, **所有代表一字不差** | 跨Agent共享缓存, 最大收益来源 |
| L1 席位固定段 | 人格卡+公开/私密信息+权力清单(含honesty映射) | 每席位固定 | 首调后永久命中 |
| L2 纪元摘要段 | 书记摘要(本纪元冻结版) | 每纪元一变 | 纪元内命中 |
| L3 追加事件段 | 本纪元内该视角可见事件, **只追加不删除** | 每次调用尾部生长 | 历史部分命中, 只有新事件全价 |
| L4 任务段 | 当前阶段+故事时间+本次任务+指定用哪个schema | 每次都变 | 全价(刻意做短, <0.5k) |

要点:

- G段把schema说明一次性全给, L4只写"本次按schema X输出"——避免每次重复大段格式文本;
- "当前故事时间/阶段"这类易变信息**禁止**出现在L4之前;
- 主席团Agent同构(G段换成主席团规则版, L1换职责+危机弧线);
- 闭门`quick_decide`、投票等小任务复用同一套G+L1+L2+L3前缀, 只换L4, 几乎纯缓存价.

## 3. 纪元(epoch)机制

把"缓存失效"与"摘要更新"绑定在同一时刻, 一次失效吃两件事:

- **纪元内**: L2冻结, L3只追加. 同一Agent每次调用 = 上次prompt + 尾部新事件 → 前缀几乎全命中;
- **纪元切换**: 当某视角的L3累积超过阈值(默认3k token), 书记将本期L3压缩为一章**追加**到L2尾部(章节追加模型, 见05§3.4), L3清空重积——L2也只从尾部生长, miss范围仅从新章节起; 低频合并(squash)时才发生L2整体重写;
- 书记摘要的触发时机**完全由纪元驱动**(废弃早期"每20条事件"方案);
- 经济学权衡: 纪元越长命中率越高但单次请求token越大, 反之亦然. 阈值为配置项`engine.epoch_l3_max_tokens`, 用实测命中率曲线调优;
- 每个视角(席位/主席团)独立纪元, 互不影响.

## 4. 事件渲染确定性纪律

同一事件任何时候渲染进prompt必须**字节级相同**: 固定模板、固定字段顺序、不掺渲染时刻的时间戳、不做随请求变化的编号重排. 否则L3"历史部分"悄悄变化, 缓存静默失效——功能上看不出来, 只有账单知道.

约束: 渲染实现为纯函数`render(event) -> str`, 配golden测试(同一事件渲染两次逐字节比对); 修改渲染模板视为破坏性变更, 会导致全场一次性miss, 只允许随版本发布.

## 5. 观测: 命中率是一等公民指标

- API响应中的`prompt_cache_hit_tokens`/`prompt_cache_miss_tokens`记入`llm_usage`表(见03§6);
- 推演界面用量面板按role展示: token消耗、费用估算、**缓存命中率曲线**;
- 命中率突然下跌 = 有代码破坏了前缀稳定性, 按bug处理. 这是比事后审计有效得多的回归防线.

## 6. 冷启动与预热

- 缓存需首个请求完成后才建立: unmod首轮各组并发时, 各Agent首调必miss(一次性成本, 接受);
- **G段预热**: 会话启动时发一次廉价请求(G段+极短任务)建立G段缓存, 之后所有代表的首调至少命中G段;
- provider无缓存能力时(如某些本地部署), 本设计无副作用——只是拿不到折扣, 不需要分支逻辑.
